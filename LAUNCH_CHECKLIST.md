# GoCheckMySports: launch checklist

Everything is built. This is what YOU need to do to go live. Roughly in order. Nothing here
requires more code from me; these are accounts, keys, and clicks.

## 1. Deploy the website (about 15 minutes)

The site is a static folder generated by `site_build.py`, and this repo (gocheckmysports) IS
the project, so the deploy is genuinely one click: the root `netlify.toml` supplies the build
command, the publish directory (`site/publish`), and headers.

1. Netlify -> Add new site -> Import from Git -> pick the gocheckmysports repo.
2. Change nothing (all settings come from `netlify.toml`).
3. Deploy. You will get a `something.netlify.app` URL, and every push to main redeploys.

Prefer drag-and-drop instead? Run `python3 site_build.py` locally and drag the
`site/publish/` folder onto Netlify. (You will re-drag after each new brief.)

## 2. Point a domain at it

The domain is `gocheckmysports.com` (the GoCheckMy family name; matches the siblings and
gocheckmy.com). Add it to the Netlify site under Domain settings; Netlify issues the HTTPS
certificate automatically. Brand strings are centralized at the top of `site_build.py`:
`NAME` (GoCheckMySports), `SLOGAN` ("The score is a fact. The story gets checked."),
`DESK_LINE`, and `FAMILY`.

## 2b. Turn on analytics (optional, family standard)

Create a Cloudflare Web Analytics site for gocheckmysports.com (same as the other GoCheckMy
sites), paste its token into `CF_ANALYTICS_TOKEN` at the top of `site_build.py`, and rebuild.
While the token is empty, no beacon is emitted at all. (The token ships empty on this desk;
the crypto chassis token was removed at cloning so sports traffic never reports to the
crypto property.)

## 2c. Replace the inherited brand art

`site/assets/og-image.png` (the social card) and `site/assets/apple-touch-icon.png` are
still the crypto chassis art. Replace both with GoCheckMySports art per
FAMILY-BADGE-LOGO-SPEC (the SVG logo and favicon are already retinted to the field green).

## 3. Turn on the newsletter (the owned audience, build this first)

The signup form on the site uses Netlify Forms, which needs zero backend:

1. In the Netlify site, enable Forms (Site settings -> Forms). Submissions show up there and
   can email you on each signup.
2. When you outgrow that, export the list to a real ESP (Beehiiv, Substack, ConvertKit,
   Mailchimp). To send from that ESP instead, replace the `<form>` action in `newsletter()`
   in `site_build.py` with your provider's embed, and rebuild.

## 4. Add the Anthropic API key so the daily brief can run

The brief (aggregate -> editor -> verifier -> writer -> digest) needs a Claude API key:

1. Get a key at console.anthropic.com and set a spend limit (the pipeline also caps per-run
   token/USD spend in `config.json`, but set an account cap too).
2. In the GitHub repo: Settings -> Secrets and variables -> Actions -> add `ANTHROPIC_API_KEY`.
3. The workflow `.github/workflows/sports-news-brief.yml` runs on its schedule and uploads
   the review queue as a downloadable artifact. Until the key is set it just runs the
   offline canary and stays green.

## 5. Your daily (or 2-3x/week) 20 minutes

This is the only recurring human cost. Pick a cadence you will actually keep.

1. Trigger the brief (it runs on schedule, or run it yourself):
   `python3 run.py --mode live`
   (or download the artifact from the GitHub Actions run).
2. Read `out/review_queue/<date>.md` (or `.html`). It shows the ranked, verified,
   drafted stories, the junk that was cut, and any editor/verifier disagreement.
3. Copy `out/approval_template.json` to `out/approval.json`. For each story you sign off,
   set `decision` to `approve` and write your take in `human_take`. Kill anything you do
   not want.
4. Publish the approved set:  `python3 publish.py`
5. Put them on the site:  `python3 site_build.py --ingest`
   then commit the new files in `site/content/` and push (Netlify redeploys), or
   re-drag the `publish/` folder.

That is it. The machine did the reading, triage, fact-check, and drafting. You did the
judgment and the voice.

## 6. Sources (all wired, all free and keyless in v1)

The desk's intake is entirely free and keyless, verified live 2026-07-19:

- **Official league data (primary tier):** MLB StatsAPI (statsapi.mlb.com) and the NHL API
  (api-web.nhle.com) are league-published ground truth for schedules, scores, and
  transactions. ESPN's scoreboard JSON (site.api.espn.com) rides along as
  stable-but-unofficial; a fetch failure there is a documented skip, never a pipeline
  failure.
- **Outlet RSS (major tier):** ESPN top lines + the NFL/NBA/MLB/NHL league feeds, BBC
  Sport, CBS Sports, Guardian Sport.
- **Aggregators (mixed tier):** Yahoo Sports. Never publishable alone; the two-source rule
  applies to anything sourced below primary.

Upgrade path if the desk ever needs deeper stats: API-Sports (~$10/mo) or SportsDataIO,
deliberately excluded from v1 under the free-only constraint.

Other switches:

- **Social auto-post / real newsletter send / video script handoff.** `publish.py` has
  adapter slots for newsletter / site / social / video. They are dry-run (they log the
  payload they would send) until you enable a target in `config.json -> publish.targets`
  and add its endpoint and credential. Wire the newsletter one first; it is the
  highest-value owned channel.
- **Tune the desk over time.** `shill_rules.json` (the hype/promotion tells + source
  reputation) and the prompts in `prompts/` are living files. Refine them as you see what
  slips through. Run `python3 verify_pipeline.py canary` after any change; it blocks on
  drift.

## The one rule, restated

Never remove the human gate to scale faster. An unsupervised sports news bot that publishes
a false injury report or a wrong score is brand-ending and a liability. Your sign-off is
the product.
